第 28 章 代码分析

28.1. 代码分析

大体分成三方面

28.1.1. 静态检测

checkstyle检查源代码的编码风格，检查javadoc是否完整，缩进，空格是否一致。用来检查大家是否遵循同一种编码规范。

pmd和cpd检测无用和重复的代码。

javaNCSS统计有效代码行数。

taglist统计代码中的TODO标记。

28.1.2. 动态检测

findbugs是检查编译后的class是否存在隐性bug，包括无用代码，不好的实践等等。

28.1.3. 关联分析

cobertura用来统计测试覆盖率。

jdepend用来检验代码依赖和复杂度。报表中可以显示当前代码都与哪些包关联。

还有把这些分析都制作成报表的dashboard，生成项目依赖图的overview，统计svn的statSVN。

28.2. checkstyle

http://checkstyle.sf.net

Checkstyle用来检测编码规范，主要用于静态检测

默认情况下使用的是sun指定的编码规范，也可以指定自己的编码规则

28.3. PMD and CPD

http://pmd.sf.net

PMD用来检测代码缺陷，比如可能出现的问题，没有使用的代码，需要优化的代码，重复的条件，重复的代码

CPD（the Copy/Paste Detector）可以列出文件中使用“复制、粘贴”造成的重复代码

两者主要用于静态检测，pmd完全没有findbugs好用，所以检测代码缺陷基本还是用findbugs，主要使用CPD统计重复代码

28.4. javaNCSS

http://www.kclee.de/clemens/java/javancss/

javaNCSS(Non Commenting Source Statements )

JavaNCSS可以计算出java文件中的有效代码行数，顺便也算出类的数量，方法的数量，注释的数量，注释的行数

28.5. taglist

http://mojo.codehaus.org/taglist-maven-plugin/

Taglist是一个maven2的插件，不能独立使用

Taglist可以统计代码中@todo,TODO,fixme等标签的数量，并显示对应的注释说明

28.6. findbugs

http://findbugs.sf.net

Findbugs可以用于检测class的缺陷，它属于动态检测，可以在官方网站看到findbugs 支持的缺陷列表

经常使用findbugs可以提高自己书写代码的质量

28.7. cobertura

http://cobertura.sf.net

Cobertura可以计算代码的测试覆盖率，让我们知道还有哪些代码是没测试过的

感觉功能与clover已经很接近了，而且配置也更加方便。

28.8. jdepend

http://clarkware.com/software/JDepend.html

Jdepend可以计算package的抽象度和依赖度

这个东西暂时还不会看，它的报表了有一大堆的统计数字

据说一个理想的包是：完全抽象的（A=1），非常稳定的（I=0），这时D=0；或者是：完全具体类构成的包（A=0），非常不稳定的（I=1），这时同样也有D=0。D=0说明包的抽象程度与稳定程度是平衡的，反之D=1说明包的平衡程度被严重破坏。

28.9. JDepend：管理代码依赖性（待研究）

JDepend遍历所有的Java代码目录，自动生成每个Package的依赖性度量。对于可扩展性、可重用性和可管理性，JDepend可自动度量一个设计在以上三个方面的质量。

JDepend为每个Package自动生成的依赖性度量指标，包括：

  1. Number of Classes (Cc)

    被分析package的具体和抽象类（和接口）的数量，用于衡量package的可扩展性。

  2. Afferent Couplings (Ca)

    依赖于被分析package的其他package的数量，用于衡量pacakge的职责。

  3. Efferent Couplings (Ce)

    被分析package的类所依赖的其他package的数量，用于衡量package的独立性。

  4. Abstractness (A)

    被分析package中的抽象类和接口与所在package所有类数量的比例，取值范围为0－1。

  5. Instability (I)

    I＝Ce/（Ce＋Ca），用于衡量package的不稳定性，取值范围为0－1。I＝0表示最稳定，I＝1表示最不稳定。

  6. Distance (D)

    被分析package和理想曲线A＋I＝1的垂直距离，用于衡量package在稳定性和抽象性之间的平衡。理想的package要么完全是抽象类和稳定（x＝0，y＝1），要么完全是具体类和不稳定（x＝1，y＝0）。

    取值范围为0－1，D＝0表示完全符合理想标准，D＝1表示package最大程度地偏离了理想标准。

为什么使用JDepend

  1. 评价设计质量

  2. 翻转依赖性

  3. 支持并行开发和极限编程

  4. 独立的发布模块

  5. 识别package的循环依赖

28.10. 编码规范与代码质量检查

28.10.1. Java编码规范

基于sun定义的Java编码规范

28.10.2. Eclipse代码格式与规范

统一Code format,Cleanup,Warning规则

导入cleanup-profile.xml和formatter.xml 到eclipse中。

经常性执行cleaup命令执行规则(对源码目录右键->source->cleanup)

28.10.3. Checkstyle

安装Eclipse-CS插件，导入springside3.0/tools/configures/checkstyle/springside_checks.xml ，并设为Default。

经常性的对项目执行checkstyle check，并在修改完毕后执行clean清除无需修改的提示。

28.10.4. PMD

安装PMD插件，导入ruleset.xml

经常性的对项目执行PMD check，并在修改完毕后执行clean清除无需修改的提示。
